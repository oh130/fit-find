[알려진 한계 — 명세서 통과에 영향 없음]
────────────────────────────

1. 추천 latency 풀 데이터셋 미검증
   - dev 모드 p95 = 187.61ms (목표 200ms 대비 여유 12ms)
   - 테스트는 소규모 인덱스 기준이며 풀 데이터(~105,000 상품)에서 재측정 필요
   - 검색 latency는 FAISS 특성상 큰 변화 없을 것으로 예상
   - 추천 Two-Tower 후보 생성 단계가 데이터 규모에 민감할 수 있음
   - 조치: 풀 데이터셋 docker-compose up 후 rec-models latency 재측정 권고

2. 학습 데이터 페르소나 불균형 (데이터 파이프라인 한계)
   - 이벤트 기준 분포: value 38% / careful 31% / trendsetter 24% / impulse 6%
   - brand_loyal 11건 / repeat_stable 15건 / color_focus 38건 / category_focus 53건 / practical 4건
   - 원인: H&M이 저가 패션 브랜드라 고객 대부분이 value 성향으로 측정됨
           희귀 페르소나 공식(brand_loyal=한부서집중, color_focus=한색상집중)이
           H&M 고객에서 극단적 패턴 요구 → 거의 해당자 없음
   - 검색/추천 시연에서는 4개 주류 페르소나 행동으로 보여주면 문제없음
   - 기능 C(온보딩) 구현 시 희귀 페르소나 유저 추천 품질은 제한될 수 있음
   - 해결하려면: data_pipeline/build_user_persona_scores.py 공식 조정
                 희귀 페르소나 조건 완화 + value 조건 강화 필요 (데이터파이프라인 담당)

────────────────────────────
[추가 기능 구현 계획] — 명세서 외 차별화 기능 (구현 순서대로)
────────────────────────────

A. 추천 기준 우선순위 조절 (1순위)
   파트별 작업:
   - api_gateway      : /api/recommend에 price_weight, popularity_weight 등
                        쿼리 파라미터 추가 → rec-models 호출 시 그대로 전달
   - rec-models       : 파라미터 받아서 rerank_bridge.py 가중치에 반영
   - frontend         : 슬라이더 UI + 파라미터 전송
   - 나머지 파트      : 무관
   - 난이도           : 낮음 / 팀 간 조율 최소

B. 추천 이유 및 활용도 설명 (2순위)
   파트별 작업:
   - api_gateway      : rec-models 응답 후 LLM API 호출
                        프롬프트: 상품명 + 카테고리 + score + reason 포함
                        응답에 reason_text 필드 추가해서 반환
   - frontend         : 추천 카드에 reason_text 표시
   - rec-models/search-engine/데이터파이프라인 : 무관
   - 사전 조건        : LLM API 키 필요
   - 난이도           : 중간

C. LLM 기반 콜드 스타트 페르소나 생성 (3순위)
   파트별 작업:
   - api_gateway      : /api/onboarding 엔드포인트 신규 추가
                        유저 자유 입력 + 선택지 → LLM 프롬프트 구성
                        LLM 응답(JSON) → session_interest 형태로 Redis 저장
                        → rec-models가 다음 /recommend 호출 시 즉시 반영
   - frontend         : 온보딩 화면 UI (자유 입력 + 선택지) 추가
   - 나머지 파트      : 무관 / 시뮬레이터 연동 불필요 (시연에서 직접 입력)
   - 사전 조건        : LLM API 키 필요
   - 난이도           : 중간~높음
   - 주의             : 희귀 페르소나(brand_loyal 등)로 온보딩된 유저는
                        학습 데이터 부족으로 추천 품질 제한됨 (위 한계 2번 참고)

D. 예산 기반 패션 세트 추천 (4순위)
   파트별 작업:
   - search-engine    : 아이템 간 CLIP 교차 유사도 API 신규 추가
                        POST /cross-similarity: article_id 리스트 → 유사도 행렬 반환
   - api_gateway      : 예산 분배 로직 + search-engine 신규 API 호출
                        예산 이내 어울리는 조합 3세트 구성 후 반환
   - frontend         : 예산 입력 UI + 세트 결과 표시
   - 사전 조건        : search-engine 팀과 API 스펙 사전 합의 필요
   - 난이도           : 높음
